# AI Collaboration Guide

이 저장소는 사람, GPT, Codex가 GitHub를 경유해 함께 작업하는 방식을 기준으로 운영합니다.

## 역할

### 사람

- 최종 방향과 우선순위를 정합니다.
- 기능 추가, 버그 수정, 기획 변경을 Issue로 남깁니다.
- Pull Request 내용을 확인하고 병합 여부를 결정합니다.

### GPT

- 기획, UX, 밸런스, 문서, 테스트 관점에서 제안합니다.
- Issue를 만들 때 목적, 완료 기준, 사용자 경험을 명확히 정리합니다.
- Pull Request를 리뷰할 때 코드뿐 아니라 게임 콘셉트와 플레이 감정도 함께 봅니다.

### Codex

- Issue 또는 명확한 요청을 기준으로 코드를 수정합니다.
- 새 장르 작업은 Godot 기준으로 구현하고, 기존 `my little boat` 작업은 Unity 기준으로 유지합니다.
- 대상 엔진에 맞는 실행 가능성을 확인하고, 변경 내용을 커밋/푸시합니다.
- README, 테스트 체크리스트, 구현 메모를 함께 갱신합니다.

## 작업 흐름

1. 사람 또는 GPT가 Issue를 작성합니다.
2. Issue에는 목표, 배경, 완료 기준, 테스트 방법을 적습니다.
3. Codex가 해당 Issue를 기준으로 브랜치 또는 커밋을 만듭니다.
4. 변경 사항은 Pull Request 또는 main 커밋으로 남깁니다.
5. GPT가 리뷰할 때는 PR 설명과 변경 파일을 기준으로 의견을 남깁니다.
6. 사람이 최종 확인 후 다음 작업을 지정합니다.

## Issue 작성 원칙

- 한 Issue에는 한 가지 목표만 적습니다.
- "좋게 만들어줘"보다 "감상모드에서 HUD 투명도를 낮추고, 다시 누르면 복구"처럼 결과를 구체화합니다.
- 대상 엔진을 반드시 적습니다. 새 장르 작업은 Godot, 기존 `my little boat` 작업은 Unity입니다.
- Godot 또는 Unity에서 확인할 테스트 방법을 반드시 적습니다.
- 제외할 것도 적습니다. 예: 결제, 광고, 온라인 기능 제외.

## Pull Request 작성 원칙

- 변경한 기능을 짧게 요약합니다.
- 대상 엔진과 확인한 장면을 적습니다.
- 아직 확인하지 못한 위험을 숨기지 않습니다.
- GPT 리뷰를 요청할 때는 "버그 가능성", "UX 어색함", "초보자 이해 가능성" 중 무엇을 봐야 하는지 적습니다.

## GPT에게 줄 수 있는 요청 예시

```text
이 저장소의 README와 현재 Issue를 보고 다음 MVP 개선안을 Issue 형식으로 정리해줘.
전투, 실패, 결제, 광고는 제외하고 힐링 표류 경험을 유지해줘.
```

```text
이 PR을 리뷰해줘.
Godot 씬 구조, 모바일 세로 UI, 초보자가 이해할 수 있는 코드 구조를 중심으로 봐줘.
```

## Codex에게 줄 수 있는 요청 예시

```text
Issue #3을 구현해줘.
Godot에서 실행할 수 있게 만들고 README 테스트 체크리스트도 갱신해줘.
```

```text
GPT가 남긴 PR 리뷰 의견을 반영해줘.
기능 범위를 넓히지 말고 지적된 부분만 고쳐줘.
```

## 현재 프로젝트의 고정 방향

- 제목: `my little boat`
- 장르: 1인칭 힐링 표류 게임
- 현재 엔진: Unity MVP 보존
- 앞으로의 새 프로젝트 기본 엔진: Godot 4.x
- Godot 우선 장르: 비주얼노벨, 뱀서라이크, 타워디펜스
- 핵심 감정: 조용함, 안정감, 부드러운 발견
- 금지: 전투, 실패, 경쟁, 결제, 광고, 온라인 편지 공유
- 우선 플랫폼: 모바일 세로 화면, PC 마우스 입력 대응

## Godot 전환 원칙

- Unity 프로젝트 루트는 그대로 유지합니다.
- Godot 프로토타입은 `godot-prototypes/` 아래에 장르별 폴더로 추가합니다.
- 하나의 프로토타입이 본격 프로젝트가 되면 별도 GitHub 저장소로 분리하는 것을 권장합니다.
- Godot 작업은 GDScript와 단순한 씬 구조를 우선 사용합니다.
- 온라인, 결제, 광고, 유료 에셋 의존은 MVP 단계에서 제외합니다.
